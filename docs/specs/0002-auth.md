# 0002. InsForge authentication

**Date**: 2026-07-17
**Status**: Proposed

## Summary

Add Google and GitHub sign in through InsForge. Keep OAuth initiation, code exchange, refresh tokens, and sign out on the server. Protect private pages before they render and match the existing HiredPilot visual system on the login page.

## Context

HiredPilot needs a trusted identity boundary before profile, job, and dashboard data can be built. The application uses Next.js 16, whose request interception entry point is `proxy.ts`, and InsForge owns authentication.

The browser must never receive a management key or refresh token. OAuth uses PKCE, which binds the callback to the browser that started the flow. Authentication and authorization remain separate: refreshing a session does not prove that a user may render a protected page.

This feature does not create a HiredPilot profile record. Database tables and row level policies belong to Feature 04.

## Requirements

**User stories**:

* As a visitor, I want to sign in with Google or GitHub so that I can use HiredPilot without creating another password.
* As a signed in user, I want my session to survive normal navigation and refresh so that I can use protected pages safely.
* As a signed in user, I want to sign out so that another person cannot continue using my session.

**Acceptance criteria**:

* **AC-1**: `/login` matches the existing HiredPilot logo, tokens, typography, card, button, focus, and responsive patterns and offers Google and GitHub controls only.
* **AC-2**: Each OAuth control starts a server side InsForge PKCE flow, stores the verifier in a short lived secure cookie, and redirects to the provider without returning auth tokens to a Client Component.
* **AC-3**: `/api/auth/callback` validates the callback code and verifier, exchanges them through InsForge, deletes the verifier cookie, and redirects a successful user to `/dashboard`.
* **AC-4**: The InsForge refresh route and Next.js 16 proxy refresh session cookies before protected Server Components render.
* **AC-5**: `/dashboard`, `/profile`, `/find-jobs`, and every `/find-jobs/[id]` page verify the current user at a server boundary and redirect unauthenticated requests to `/login`.
* **AC-6**: Authenticated requests to `/` and `/login` redirect to `/dashboard`; the authenticated dashboard shell provides a navbar action that signs out through InsForge, clears the session, and redirects to `/`.
* **AC-7**: Cancelled OAuth, invalid callbacks, missing verifier cookies, refresh failure, and sign out failure produce safe behavior and human readable messages without exposing raw provider errors.
* **AC-8**: Auth code uses current Next.js 16 and `@insforge/sdk` APIs, passes lint and the production build, and adds no forbidden color values or raw Tailwind color classes.

## Options considered

### Option 1: InsForge SSR auth with server initiated OAuth

Use the SSR helpers exported by `@insforge/sdk`. Server Actions initiate OAuth and sign out, a Route Handler exchanges the OAuth code, and `proxy.ts` refreshes cookies.

**Pros**:

* Keeps access and refresh handling at the server boundary.
* Reuses the backend already chosen for HiredPilot.
* Supports protected Server Components without a second auth provider.

**Cons**:

* Requires several small framework integration files.
* Depends on an InsForge SSR contract whose framework specific MCP page is currently missing.

### Option 2: Browser only InsForge auth

Call `signInWithOAuth()` from a Client Component and let the browser SDK detect the callback.

**Pros**:

* Requires fewer server files.
* Matches the simplest generic SDK example.

**Cons**:

* Does not provide the required server owned session boundary for protected Server Components.
* Makes callback and token behavior harder to reason about across server rendering.

### Option 3: Add a second hosted auth provider

Use a separate identity service and bridge its tokens into InsForge.

**Pros**:

* Could provide a broader set of prebuilt identity screens.

**Cons**:

* Adds another vendor, token model, dependency, and operational surface without a product requirement.

## Decision

**Chosen option**: Option 1: InsForge SSR auth with server initiated OAuth

Use the current `@insforge/sdk/ssr` and `@insforge/sdk/ssr/middleware` exports. Keep OAuth mutation and callback work on the server. Use browser clients only to consume an existing session when a later feature needs client auth state.

## Rationale

The application already depends on InsForge, so its SSR helpers are the smallest secure fit. Browser only auth is attractive for a static client application, but HiredPilot needs protected Server Components and server scoped database access. A second provider would duplicate identity concerns without solving a current requirement.

## Feature design

**Data model sketch**:

* `InsForgeUser`: provider owned user with required `id`, `email`, and provider metadata.
* `InsForgeSession`: provider owned access and refresh state. The application stores it only through SDK managed cookies.
* `OAuthAttempt`: transient state represented by the required `insforge_code_verifier` HTTP only cookie. It expires after ten minutes and is deleted after callback handling.
* No HiredPilot database entity or migration is created in this feature.

**State transitions**:

`signed_out` to `oauth_pending` to `signed_in` to `refreshing` to `signed_in`. A callback failure returns to `signed_out`. Sign out or an unrecoverable refresh failure returns to `signed_out`.

**API surface**:

| Surface | Method | Key inputs | Key outputs | Auth | Key errors |
|---|---|---|---|---|---|
| `startOAuth` Server Action | action | `provider: "google" \| "github"` | Redirect to provider | Public | Unsupported provider, missing OAuth URL, provider error |
| `/api/auth/callback` | GET | `insforge_code`, verifier cookie | Session cookies, redirect | Public callback | Missing code, missing verifier, exchange failure |
| `/api/auth/refresh` | POST | SDK cookies and CSRF data | Refreshed session cookies | Session cookie | Expired or invalid refresh state |
| `signOut` Server Action | action | SDK cookies | Cleared session, redirect | Authenticated | Provider sign out failure |
| `requireUser` server helper | function | Request cookies | Current InsForge user | Authenticated | Missing or invalid session |
| `proxy.ts` | proxy | Request and cookies | Refreshed response or redirect | Mixed | Refresh failure |

**Key invariants**:

* Only `google` and `github` are valid OAuth provider inputs.
* OAuth starts and finishes on the server.
* `insforge_code_verifier` is HTTP only, secure in production, same site lax, scoped to the application, and short lived.
* Refresh and authorization are separate checks.
* Protected pages verify a current user before rendering.
* Auth errors shown to users are stable application messages, never raw provider text.
* No management key or refresh token enters browser code.

**Security model**:

Public users may open `/`, `/login`, the callback route, and the refresh route. Only an authenticated InsForge user may render protected routes or invoke sign out. There are no roles or team accounts. Future user owned data access must use the verified user ID and row level policies from Feature 04.

**Configuration required**:

* `NEXT_PUBLIC_INSFORGE_URL`: browser safe InsForge backend URL.
* `NEXT_PUBLIC_INSFORGE_ANON_KEY`: browser safe anonymous credential.
* `APP_URL`: trusted absolute application origin used to build the OAuth callback URL.
* InsForge dashboard configuration for Google and GitHub credentials and allowed callback URLs.

**Critical test scenarios**:

* Happy path: Google and GitHub each complete OAuth and land on `/dashboard`, verifies **AC-1**, **AC-2**, and **AC-3**.
* Session path: a valid session survives navigation and refresh while protected pages render, verifies **AC-4** and **AC-5**.
* Failure case: a cancelled or invalid OAuth callback returns to `/login` with a safe retry message and removes transient state, verifies **AC-7**.
* Auth case: a signed out request to every protected route redirects before page content renders, verifies **AC-5**.
* Sign out case: the navbar action ends the session and protected pages become unavailable, verifies **AC-6**.
* Standards case: lint, build, source review, and token review pass, verifies **AC-8**.

## Build plan

1. Build the login page shell with the existing HiredPilot visual system and accessible provider controls, satisfies **AC-1** and **AC-8**.
2. Install the approved InsForge SDK and add typed environment, server client, and current user helpers, satisfies **AC-4**, **AC-5**, and **AC-8**.
3. Add server initiated Google and GitHub OAuth plus the callback exchange, completing one end to end sign in journey, satisfies **AC-2**, **AC-3**, and **AC-7**.
4. Add the refresh route and Next.js 16 proxy, then enforce protected server boundaries, satisfies **AC-4**, **AC-5**, and **AC-7**.
5. Add the authenticated dashboard shell, navbar sign out action, and public route redirects for authenticated users, satisfies **AC-6** and **AC-7**.
6. Verify both providers, callback failure, refresh behavior, route protection, sign out, responsive UI, lint, and production build, satisfies **AC-1** through **AC-8**.
7. Record the completed auth surfaces in the UI registry and progress tracker, satisfies **AC-1** and **AC-8**.

## Consequences

**Positive**:

* One provider owns identity, session, and later data authorization.
* Protected Server Components receive a verified user before reading private data.
* OAuth secrets and refresh state stay out of client components.

**Negative / tradeoffs**:

* Auth depends on InsForge service availability and correct provider dashboard settings.
* The integration has more server files than the generic browser only example.
* Local OAuth verification needs valid provider credentials and callback configuration.

**Neutral**:

* Profile creation remains deferred to the database schema and profile features.
* PostHog identify and reset calls remain deferred to Feature 03.
